# Задача
Итак, перед нами задача преобразовать какой-то текст, подчиняющийся заданному синтаксису в абстрактное семантическое дерево (далее ast). Предположим что у нас есть файл с выдуманным языком программирования. Как нам его распарсить? Начнём с мысленного анализа. Давайте просто посмотрим на воображаемый код. Мы можем разбить его на отдельные логические зоны. Зона объявления глобальных переменных, зона импортов, зона функций. Можно выделить и подзоны. Зона параметров функции и зона с кодом функции. Эти зоны могут синтаксически очень сильно отличаться. Вот пример псевдокода
``` go
import (
  alfa
  betta
  gamma
)
fun foo(a, b) {
  Hello.world();
}
```
После импорта нет точки с запятой, в функции есть; В импорте нет запятых, в параметрах функции есть. По этой причине хотелось бы разбить весь файл на зоны, а затем, в зависимости от зон, группировать символы в какие-то небольшие смысловые блоки. При этом логические блоки в разных зонах будут разные, так как разные зоны подчиняются разным синтаксическим правилам.
Вот и вырисовался какой-никакой алгоритм действий. Но если бы всё было так легко, я бы не писал это. Проблема в том, что мы не придумали, как именно файл разбить на зоны. И дело даже не в технической реализации. От изменения всего одного символа в файле может измениться всё. Представьте, что вы забыли закрыть кавычку. Помните что обычно делала подсветка синтаксиса? Половина всего написанного кода превращалась в строку. Приведу ещё один, уже реальный пример из js.
``` js
() => { /*...*/ } // 1
() => ({ /*...*/ }) // 2
{ /*...*/ } // 3
a = { /*...*/ } // 4
```
Здесь подзона в фигурных скобках очень сильно зависит от контекста. В случаях 1 и 3 это зона с кодом, содержащая какие команды. В случаях 2 и 4 это инициализация объекта, совершенно отличающиеся по синтаксису и семантике.
Получается что зоны очень сильно зависят от отдельных символов или их комбинаций. А эти отдельные символы, это те самые смысловые блоки, для нахождения которых нам нужно было разбить файл на зоны. Замкнутый круг, но sts решил эту проблему.
В sts мы предлагаем делать всё сразу: парсить токены (те самые смысловые блоки) и находу определять зоны. Комбинированный подход. Сейчас разберёмся с базой языка, а более интересное, постороение нормального, независимого от синтаксиса ast я расскажу во второй части.
# Глава 1: База STS
## Главная зона
Для начала нам нужно объявить главную зону, с которой начинается любой файл.
``` java
--main_zone--
```
Теперь стек зон начинается с `main_zone`. (Вы можете задать любое другое название) В дальнейшем мы сможем добавлять в стек ещё зоны. Верхний, последний элемент стека, это текущая зона. Стек обеспечивает что-то вроде рекурсии контекста. Условный пример: когда парсер будет на строке внутри условия внутри цикла внутри функции внутри класса внутри массива классов внутри функции `main`, стек будет выглядеть примерно так:
``` yaml
if_body # верхний элемент – текущая зона
for_body
fun_body
class_body
array_zone
fun_body
main_zone # нижний элемент – первая, главная зона
```
Когда стек будет полностью очищен, мы будем считать, что парсинг завершён.
## Токены
Токены, очень важная часть парсера, ведь на них лежит ответственность за распознавание комбинаций символов. Для этого в токенах используются особые текстовые выражения. Обозначаются как `regex`, но не путайте с регулярными выражениями, наши текстовые выражения лишь вдохновлялись regex'ом.
Давайте сделаем токен для условной конструкции:
``` java
IF {
  parentZone: main_zone;
  regex: "if";
}
```
По коду:
- `IF` – имя токена. Оно может быть практически любым, так как в sts практически нет ключевых слов.
- `parentZone: main_zone;` – здесь мы указываем, в какой зоне данный токен стоит искать. То есть токен `IF` пренадлежит зоне `main_zone`. Когда текущая зона будет `main_zone`, или её заимствующая (про это позже), то парсер будет проверять токен на совпадение. Также можно написать `parentZone: a, b, c;`, чтобы сделать сразу несколько зон имели токен.
- `regex: "if";` – Регулярное выражение для поиска токена условной конструкции. Последовательность двух символов `if` будет определяться как токен `IF`.
### openZone
Теперь надо разобраться с ещё одной частью. Как добавить зону в стек, тем самым перейдя в подзону? Для этого используется команда  `openZone`. В часности, у токенов есть параметр `openZone`, который открывает новую зону, из текущего токена, добавляя её в стек.
``` java
BODY {
  parentZone: main_zone;
  regex: "\{";
  openZone: body_zone;
}
```
Теперь, когда парсер встретит фигурную скобку в главной зоне, он создаст токен `BODY`, и откроет зону `body_zone`. При этом экземпляр токен `BODY` будет указан владельцем им открытой зоны `body_zone`, потому в дальнейшем, все токены, которые будут найдены в открытой зоне, будут считаться детьми их родительского токена `BODY`, и добавятся в его переменную `children`.
### setZone
Кроме `openZone` есть и `setZone`, который заменяет текущую зону, вместо добавления новой на верхушку стека. Это может быть полезно, когда нам нужно токены из нескольки зон поместить в один список children или задать особую последовательность гаризонтального синтакиса. // TODO: перенести во вторую часть
## Зоны
Кроме токенов у нас есть зоны. Их тоже можно объявить и задать разные параметры. Объявлять зоны для использования не обязательно, хотя делает код читабельней.
``` java
- body_zone (
  < main_zone;
)
```
Самая востребованная функция зон, это заимствование. Знак `<` означает, что зона заимствует другую зону. Заимствование зоны, это перенимание её токенов. На примере выше: есть какие-то токены, у которых указано `parentZone: main_zone`. Все эти токены так-же будут искаться и в `body_zone`, так как мы написали. Если сделать ещё одну зону, которая заимствует у `body_zone`, то в ней будут токены и из `body_zone`, и из `main_zone`. Заимствовать можно сколько угодно зон.
Есть ещё один оператор. `>` наоборот, делится токенами с другой зоной. Фактически, суть не меняется, вы можете просто перенести `< main_zone` из `body_zone` в `main_zone` в виде `> body_zone`. Меняется только носитель ответственности за заимствование.
## closeZone и ghost-токен
Ещё одна команда `closeZone`. Она завершает зону, уберая её со стека, записывая все токены в переменную `children` токена, открывшего зону.
``` java
BODY_CLOSE! {
  parentZone: body_zone;
  regex: "\}";
  closeZone;
}
```
Вроде всё понятно. В `body_zone` встретив `}` мы закрываем зону. Но что значит восклицательный знак после имени токена? Это ghost-токен, который выполняет временную функцию, а сохраняться в ast не будет. Технически такой токен исчезает сразу после завершения его парсинга.
## Улучшения и промежуточный результат
Посмотрим что сейчас получается, из всего, что мы написали
``` lisp
ififif{ifif}if
```
данный код преобразуется в следующие дерево
``` yaml
children:
  -IF
  -IF
  -IF
  -BODY:
    children:
      -IF
      -IF
  -IF
```
негусто. Но важно понимать, что sts за вас правила не придумывает, а главное, он задумывался как универсальный инструмент, который можно применять для совершенно разных языков. Именно поэтому он не будет сам придумывать, как обрабатывать пробелы, новые строки, и так далее. Вам нужно прописать всё это самостоятельно. Давайте добавим ещё немного кода:
``` java
VOID_TOKEN! {
  parentZone: void_zone;
  regex: "+[ \t\r\n]";
}
- void_zone (
  > main_zone;
  > example_indefinite_zone;
)
COMMAND {
  parentZone: body_zone;
  regex: "[a-zA-Z]*[a-zA-Z1-90]";
}
```
Давайте пройдёмся по нововведениям
1. Теперь у нас есть ghost-токен пустоты, который будет обрабатывать пробелы, табы и новые строки. С ним появилась `void_zone`, которая передаёт токен в `main_zone` и `example_indefinite_zone` (добавил её тупо как пример)
2. Появился токен команды, который может быть только в `body_zone`. Регулярное выражение токена означает текст, который начинается с любой английской буквы, а затем содержит сколько угодно букв и цифр.
``` lisp
if ifif {   }{
  run make {
    ifs12
  }
}
```
Этот код преобразуется в:
``` yaml
children:
  -IF
  -IF
  -IF
  -BODY:
    children: []
  -BODY:
    children:
      -COMMAND:
        text: "run"
      -COMMAND:
        text: "make"
      -BODY:
        children:
          -IF
          -COMMAND:
            text: "s12"
```
Тут кратко нужно уточнить, что `text`, это переменная, в которую сохраняется текст из оригинального файла, который и совпал с регулярным выражением. Я опустил эту переменную у токенов `IF` и `BODY` потому что в них `text` всегда одинаковый. Чтобы помочь парсеру не тратить свои силы на эти токены, добавим к параметру `regex` восклицательный знак. Рекомендуется делать это со всеми токенами, для которых текст сохранять не нужно.
``` java
IF {
  parentZone: main_zone;
  regex!: "if";
}
BODY {
  parentZone: main_zone;
  regex!: "\{";
  openZone: body_zone;
}
```
Так text сохраняться не будет.
## text exception
Пора разобраться и с регулярками. Разбераться будем глубоко, с семантикой.
Само по себе текстовое выражение, это список упорядоченных конструкций, описывающих символьные закономерности. Если нам удаётся соотнести регулярное выражение с рядом символов из текста, то мы успешно нашли соответствие, каретка парсера сдвигается, и токен с этим regex'ом создаётся, а в его переменную `text` записывается соотнесёный текст. Теперь разберёмся, какие конструкции в рамках regex'а существуют:
#### Символ
Просто записываем символ. `regex: "if"` – содержит две конструкции символа и соответствует строке `if`.
Кроме конкретных символов есть «любой символ» обозначается как `.` и соответствует любому символу.
Всё классно, если символ не занят каким-то оператором. Все буквы и цифры 100% можно писать в чистом виде. С остальными символами могут возникать вопросы, так как каждый раз перепроверять всю документацию на наличие у этих символов каких-либо задач может быть не очень удобно, поэтому вы всегда можете экранировать (добавить обратный слеш) любой символ кроме букв и цифр. Экранирование в sts нативное, писать `"\\$"`, дублируя слеш дважды не надо. Просто `"\$"`, `"\n"`.
Несколько символов не сливаются в единый элемент выражения. Каждый является отдельной конструкцией.
#### Диапазон
Диапазон указывает множество допустимых символов. Соответствие определяется, если индекс символа находиться между индексами границ. Вы задаёте граничные символы через знак минуса. Пример: `a-z`, `2-8`
#### Группа
Круглые скобки. В них можно сгрупировать конструкции.
Прочитав чуть далее про повторы будут понятны эти примеры:
- `+hello` - повтор только `h`
- `+(hello)` - повтор всего слова `hello`
#### Вариация
В обычном regex'е квадратные скобки используются для перечесления допустимых символов, но sts пошёл дальше, в нём это только частный случай. Sts решил что в квадратных можно перечислить любые конструкции, сделав выбор одно из. Вот примеры:
- `[agz]` – один из трёх символов.
- `[a-z0-9]` – один из двух диапазонов.
- `[(\/\/)(\#)( )]` – одна из трёх групп. комбинация двух слешей, либо одна решётка, либо пробел.
У вариации есть ещё одна функция – исключения. Расположив справа от `^` конструкции, они будут считаться не допустимыми. У исключений максимальный приоритет, в не зависимости от того, что вы пишите слево.
Примеры:
- `[a-z^s]` – буквы от a да z исключая s
- `[.^0-9]` – Любой символ, проме знаков до 0 до 9. (Также в sts есть оптимизация. Если единственная конструкция в левой части вариации – любой символ, то его можно не писать: `[^0-9]`)
- `[a-zA-Z^(if)(for)]` – буква, которая не пренадлежит слову if и for. Немного сложно, но вы поймёте!
#### Повторение
указывает на повторение какой-то конструкции. Конструкция, которую вы хотите повторять ставится после оператора повтора. Самих операторов сейчас существует 4 штуки:
- `+` - обязательное повторение
- `*` - необязательное повторение
- `++` - обязательное ленивое повторение
- `**` - необязательное ленивое повторение
**ВАЖНО**: не путайте с синтаксисом регулярных выражений, в которых оператор повтора ставиться **после** конструкции. Здесь они ставятся **перед** конструкцией.
Обязательные повторения требуют, чтобы конструкция встретилась хотя бы один раз, чего обратным не требуется.
Пример:
- `+.` – один и более любой символ
- `*a` – ноль и более букв `a`
Ленивые повторения хотят как можно быстрее завершиться, что не всегда возможно, а неленивые наоборот, будут повторять до последнего, пока засчитать ещё один повтор станет уже невозможно.
Пример: у нас есть следующий текст: `0011aa`. Теперь рассмотрим два похожий выражения, дающих разные результаты, в зависимости от ленивости:
- `+(0-9)[1a]` – здесь неленивый повтор. О будет идти до последний цифры, а затем перейдёт к `[1a]`, после чего будет считана ещё одна буква. Итоговый `text`: `0011a`
- `++(0-9)[1a]` – здесь ленивый оператор. Каждый повтор он будет пытаться сказать следующей конструкции: «чувак, твоя очередь уже, ты вроде подходишь». Пока наш повтор сопоставляется с нулями, выражение `[1a]` не подходит. Но как только появляется единица, подходящая под `[1a]`, повтор завершается. Итоговый `text`: `001`
На практике это может применятся для создание длинных коментариев, которые завершаются `*/`. Вот как я составил для них текстовое выражение: `\/\***.\*\/`. Чтобы было понятней, добавлю скобки: `(\/\*)(**.)(\*\/)`. Сначало идёт открывающая конструкция `\/\*`, затем ленивый цикл любых символов, после которого идёт закрывающая конструкция `\*\/`.

#### Необезательный элемент
Префикс `?` перед конструкцией делает её не обезательной. То есть повтор 0 или 1 раз.
К примеру мы хотим написать regex для разных чисел: целые, десятичная дробь, экспоненциальная запись. 
``` regex
+0-9?(\.+0-9)?([eE]?[\+\-]+0-9)
```
#### Оператор сохранения
Для вычленения из regex'а отдельных частей наружу, можно использовать оператор сохранения. Поставив перед любой конструкцией `$`, соответствующий ей текст будет сохраняться во временую переменную токена `$n`, где n – номер символа `$` от начала регулярного выражения. Как взаимодействовать с этими переменными, подробно расскажу во второй части. Пока я думаю хватит такого примера:
``` java
COMMENT_TOKEN {
  parent_zone: main_zone;
  regex!: "\/\*$**.\*\/";
  $comment = $1; # делаю переменную comment, в которую копируется содержание комментария без /* */
}
```
#### Пост-просмотр
Расположив перед конструкцией `>` вы будете проверять наличие конструкции далее, без сдвига коретки вперёд. Думаю на примерах будет понятней.
- `if>[.^a-zA-Z]` – слово `if` после которого не идёт буква.
- `foo>[^(bar)]..r` – Очень сложный пример. Надо вдуматься. Соответствует `foo` и три буквы, которые зоканчиваются буквой `r` но не являются словом `bar`. Подойдёт `foorar`, но не подойдут `foobar` и `foogon`.
### Regex-link
Регулярные ссылки, это ещё одна структура в sts файле. Чаще всего их распологают в самом начале файла. Они позвляют привязать регулярное выражение к имени.
``` java
/name_char"[a-zA-Z0-9\_\$]"/
```
После вы можете встраивать ссылку в любоё регулярное выражение, окружив имя ссылки слешами.
``` java
regex: "+/name_char/[;\n]";
```
### Итоги текстовых выражений
В текстовые выражениях STS сильно просвечиваются регулярные выражения, но тем не менее они сильно отличаются. Здесь нельзя задать чёткое количество повторов `{2}`, но есть на много более гибкая вариация. В регулярках полвина операторов постфиксные, в то время как в STS все префиксные. Тектовые выражения были созданны под задачи STS.
Давайте дополним наш файл реальными примерами regex-link.
``` java
# первый символ имени
/nsc"[a-zA-Z\_\$]"/
# символ имени
/nc"[a-zA-Z0-9\_\$]"/
# символ НЕ имени впереди
/nn">[^a-zA-Z0-9\_\$]"/
# имя
/name"/nsc/*/nc//nn/"/
# символ пустоты
/vc"[ \t\r\n]"/
# обязательная пустота
/void"+/vc/"/
# необязательная пустота
/vvoid"*/vc/"/
# коментарии
/scomm"\/\/+[^\n][\n@e]"/
/lcomm"\/\***.\*\/"/
```
Поясню применение некоторых выражений:
`name` – везде где нужно задать имя чего-то. В зависимости от потребностей вашего языка можно изменить выражение в ссылке, оставив логику использования той же.
`nn` – будет использоваться, как обозначание конца именного токена. Стоит добавлять в любые токены с ключевыми словами. На пример выражение `for` отъест кусок от слова `force`, которое не хотело быть циклом. Вместо этого сделаем выражение `for/nn/`. Теперь оно оно будет разпозноваться только в случаях вроде `for abc`, `for()` но не `force`, `forma`. Добавив это в токен `IF`, получим следующий код: `regex: "if/nn/"`
Что косается void и пустоты, их я реализую следующим образом:
``` java
- void_zone (...)

  parentZone: void_zone;
  regex!: "/void/";
}

  parentZone: void_zone;
  regex!: "/scomm/";
}

  parentZone: void_zone;
  regex!: "/lcomm/";
}
```

Теперь у меня есть зона, способная обрабатывать весь мусор, который не должен попадать в конечный ast. Теперь остаётся эту зоны заимствовать во всех зонах, где это требуется, и тогда можно будет добавлять комментарии и пустые символы между любыми токенами.

## Приоритеты
Начну с самого острого незакрытого вопроса первой части. Как разрешаются конфликты токенов?
В стандартном случае порядок вообще ни как не гарантируется. STS может составить таблицу префиксов, а может в тупую переберать токены в порядке объявления, или в каком-то другом порядке. Когда парсер успешно матчит какой-то токен (сопоставляет текст с выражением) перебор токенов тут же останавливается, и остольные токены не проверяются.
Это не самый надёжный подход, поэтому появляется ещё один параметр: `priority`, задающий порядок перебора токенов. По умолчанию он считается равным нулю. В рамках одной зоны, первыми будут проверяться токены с большим значением приоритета. Вот пример, который будет работать не совсем классно:
``` java
# фрагмент кода
NAME {
  parentZone: code;
  regex: "/name/";
}
FOR {
  parentZone: code;
  regex: "for/nn/";
}
```
Если я попытаюсь от парсить файл, указав зону `code`, я получу достаточно не определённое поведение. Если мы предположим, что никаких оптимизаций не применяется, то мы никогда не увидим `FOR` в конечном ast, так как любой `for` будет первым съедать `NAME`. Чтобы это исправить, я бы внёс в `NAME` отрицательный приоритет, чтобы все добавляемые в будущем кейворды с нулевым приоритетом правильно работали.
``` java
NAME {
  parentZone: code;
  regex: "/name/";
  priority: -5;
}
```
Теперь мы можем без проблем добавить ещё один кейворд.
``` java
WHILE {
  parentZone: code;
  regex: "while/nn/";
}
```
Важное уточнение: приоритет не задаёт последовательность токенов, а лишь указывает, в каком порядке их проверять. Если вы напишите два токена с приоритетами 1 и 2, это не значит, что парсер вам построит дерево с `TOKEN1`, `TOKEN2` по порядку.
### Приоритеты у зон (TODO: убрать или сократить)
Чаще всего у каждой зоны своё множество токенов, не пересекающаяся с другими зонами. Когда же нам приходиться заимствовать токены, может начаться путаница с приоритетами. На такие случаи есть особые настройки, которые позволяют задать «относительность приоритетов»:
``` java
- zoneA (
  < zoneB {
    priority: -1;
  };
)
```
Здесь мы пристыковываем к оператору `<` фигурные скобки, в которых указываем параметры заимствования. `priority: -1` здесь относиться не к отдельным токенам, а ко всем сразу. Между собой все токены в `zoneB` сохраняют порядок относительно друг-друга. Так же относительный порядок сохраняют токены `zoneA`. Но присоединяющиеся из `zoneB` токены все расположатся до токенов `zoneA`, так как приоритет заимствования `-1`, т.е. меньше. 
Все токены `zoneB` идут после всех токенов `zoneA`.
## Устройство парсера
Для понимания большинства последующих тем стоит разобраться со структурой самого парсера STS.
### Структура
MetaParser – самый первый модуль, который не занимается анализом языковых файлов, а занимается предварительным парсингом .sts файлов. Тех самых, которые всё это время мы учились писать. Он же расставляет все токены в порядке приоритетов, создаёт таблицы зон и тому подобное. Затем metaparser отправляет всё в следующий модуль – contentparser.
ContentParser занимается уже основной задачей: парсинг файлов на основе созданных вами правил. На выходе получается всевозможная информация, в том числе и ast дерево, которое далее может отправиться в ваш компилятор для дальнейшего переваривания.
### ContentParser output
объект, который мы получаем от контент-парсера содержит ряд разых данных. Впервую очередь ast токенов, с которым мы будем работать 90% времени. Затем значения глобальных переменных и флагов. В силу текущих познаний, пройдёмся по ast.
### Abstract Symantic Tree
Ast напоминает структуру любого формата вроде JSON или YAML, но не имеет текстового представления, зато обладает дополнительными свойствами: ссылками, специальными системными параметрами, типизацией в некоторых местах. Все экземпляры токенов представляются объектами с параметрами вроде `text` и `children`. Также есть системный параметр `prototype`, указывающий на объявление токена.й
Давайте для разминки мозга напишем интерпретатор выдуманного безполезного змеиного языка. Сначала на js с высокоуровневой либой, которая всё красиво завёртывает под js. А потом на си, используя официпльный интерфейс STS.
В нашем япе будет `PRINT` для вывода слова в консоль, и `BODY`, хранящий в себе другие токены, и имеющий числовой параметр `loops`, из-за которого `BODY` будет повторять своё содержимое `loops` раз. Заметьте, я вообще не коснулся синтаксиса языка. Я описал только токены, с которыми мы будем работать, и в этом по моему мнению главная красота разделения синтаксиса и семантики.
``` js
let ast = sts.parse("example.anakonda", stsFile).ast;
function element(token) {
  if (token._prototype.name == "PRINT") {
    console.log(token.printText);
  }
  else if (token._prototype.name == "BODY") {
    for (let i = 0; i < token.loops; i++) {
      for (let t of token.children) {
        element(t);
      }
    }
  }
}
for (let t of ast.children) {
  element(t);
}
```
реализация на си не такая красивая, так как здесь мы работаем напрямую с сишными структурами из sts, а не с красивой оболочкой на js.
``` c
void execute(Sts_MetaFile* stsFile) {
  Sts_ContentFile* cFile = ContentParser_parse(
    "example.anakonda", 
    stsFile,
    null, // дополнительные параметры
  );
  for (int j = 0; j < cFile->ast->children.length; j++) {
    ContentToken* t = ContentTokenList_get(cFile->children, j);
    element(t);
  }
}
void element(ContentToken* token) {
  char* name = token->prototype.name;
  if (equalstr(name, "PRINT")) {
    char* text = ContentParams_get(token->params, "printText").string;
    printf("%s", text);
  }
  else if (equalstr(name, "BODY")) {
    int loops = (int) ContentParams_get(token->params, "loops").number;
    for (int i = 0; i < loops; i++) {
      for (int j = 0; j < token->children.length; j++) {
        ContentToken* t = ContentTokenList_get(token->children, j);
        element(t);
      }
    }
  }
}
```
В принципе, это уже рабочие интерпретаторы, требуется только добавить пару строк, которые делают `stsFile` из, собственно, sts файла. Но это тема не этой статьи, я лишь привёл пример интерпретатора для языка со сколь угодно глубоко вложенными конструкциями. Что не редкость для современных языков. Да даже в таких ископаемых как паскаль можно создавать циклы в циклах в циклах в циклах. Научить парсер читать синтаксис любой глубины вложенности, с учётом контекста, и есть одна из задач STS.

## Переменные
Одна из очень интересных тем, это переменные в STS. Все переменные делятся на два типа: стандрартные и созданные, и несколько видов, которые я вскоре покажу.
### Параметры
Первый вид переменных, это параметры. `text` и `children` как раз являются переменными-параметрами, при этом их создаёт парсер, вручную их создавать не надо. А чтобы создать свой параметр, напишем следующие:
``` java
PRINT {
  ...
  $[printText];
}
```
именно так в нашем языке «Анаконда» создаётся параметр `printText`. Остаётся предать этой переменной значение. Тут пора разрушить неведение перед синтаксисом, которое я специально создал во время написания интерпретатора Анаконды, и что-нибудь такое придумать.
``` java
PRINT {
  parentZone: main;
  regex!: "echo:$(*[^\n])[\n@e]";
  $[printText] = $1;
}
```
Вот и пригодился оператор сохранения. `text` мы не сохраняем, так как написали `regex!`, но временно сохранённая переменная `$1` остаётся. Но переменные, созданные оператором сохранения вида `$n` существуют только в рамках токена и исчезают, как только завершается обработка токена. Пока `$1` не растворилась, мы записываем её в нормальный параметр `printText`.
Ещё интересней, когда мы хотим из одного токена писать значения в другой. Когда вы создаёте переменную, вы пишете знак доллара, затем имя переменной в квадратных. Когда вы обращаетесь к уже существующей переменной, пишите имя без скобок. А если имени в рамках текущего токена нет, то оно будет искаться по цепочке вниз: у родителя, родителя родителя и так далее. За счёт этого и реализуется обратная связь между токенами. Давайте продолжим писать парсер япа.
``` java
BODY {
  parentZone: main;
  regex: "\>";
  openZone: body;
  $[loops] = 1;
}
- body (
  < main;
  < void_zone;
)
BODY_COUNT! {
  parentZone: body;
  regex: "+[0-9]";
  $loops = $text;
}
BODY_CLOSE! {
  parentZone: body;
  regex: ";";
  closeZone;
}
```
Здесь мы видим, что токен `BODY`, открывает зону `body`, которая расширяется зоной `main`, т.е. в зоне `body` будут все токены из `main` и её личные токены: `BODY_COUNT!` и `BODY_CLOSE!`. При этом оба призраки. С `BODY_CLOSE` всё понятно, он служит для закрытия зоны, открытой токеном `BODY`. С `BODY_COUNT` намного интереснее. В нём текстовое выражение ищет число. А затем `$loops` приравнивается к `$text`. Переменной `loops` в `BODY_COUNT` нет, а значит он пойдёт искать её вниз по дереву. А кто может иметь переменную `loops` ниже токена в зоне `body`? Только токен `BODY`, которому мы эту переменную прописали. А что на счёт `text`? А он уже есть в токене `BODY_COUNT` и содержит число (в текстовом представлении). Получается, что мы запишем в переменную `loops` родительского токена число, а затем токен `BODY_COUNT` раствориться, так как является ghost. А если токен не встретится, то переменная будет равна одному, как мы и написали.
### Параметры зон
Второй вид переменных – параметры у зон.
``` java
- hello_zone (
  $[hello_counter];
)
```
 К ним так же можно оброщаться, как и к параметрам токенов. Когда переменная ищется, зона стоит выше, чем токен открыший её. То есть поиск переменной сначало произойдёт в зоне, потом в её токене владельце, потом в родительской зоне токена, а затем в токене, открышем ту зону. Но поскольку зоны, это временные объекты в стеке, то значения их переменных стерается, как только зона закрывается.
Разберём пример. У нас есть язык, с синтаксисом объявления функций, как в java, но с ключевым словом `fn` в начале
``` java
fn public static int main(int i);
```
Мы хотим, чтобы токен опеределённого свойства не мог быть использован два раза в зоне свойств функции. Но при этом свойства можно писать в любом порядке или не писать.
``` java
// допустимо
fn public void foo();
fn static private void foo();
fn private static void foo();
fn void foo();
// не допустимо
fn public public void bar();
fn public static private void bar();
fn static static void bar();
```
Для этого я реализую следующий механизм:
``` java
- main_zone (
  < fun_zone;
)

FN {
  parentZone: fun_zone;
  regex: "fn/nn/";
  openZone: fun_property;
}

- fun_property (
  < void_zone;
  < access_modifier_zone;
  < static_modifier_zone;
);

- access_modifier_zone (
  $[am_isUsed] = false;
);

ACCESS_MOD_PUBLIC {
  parentZone: access_modifier_zone;
  regex: "public/nn/";
  $am_isUsed = $am_isUsed
    ? error("модификатор доступа уже указан")
    : true;
}
ACCES_MOD_PRIVATE {
  parentZone: access_modifier_zone;
  regex: "private/nn/";
  $am_isUsed = $am_isUsed
    ? error("модификатор доступа уже указан")
    : true;
}

- static_modifier_zone (
  $[st_isUsed] = false;
);

MOD_STATIC {
  parentZone: static_modifier_zone;
  regex: "static/nn/";
  $st_isUsed = $st_isUsed
    ? error("модификатор static уже указан")
    : true;
}
```
Мне кажется это очень красивое решение! Я использовал множество механизмов STS, которые мы раньше разбирали, и теперь мы объединяем их все воедино.
Сначало я определяю зону свойств функции, которая заимствует зоны `void_zone`, `access_modifier_zone`, `static_modifier_zone`.
- `void_zone` мы знаем, это обработка пустот.
- `access_modifier_zone` содержит два токена для `public` и `private` и переменную `am_isUsed`. Когда один из токенов парсится, и начинается этап обработки переменных, начинает выполняться этот код:
  ``` java
  $am_isUsed = $am_isUsed
    ? error("модификатор static уже указан")
    : true;
  ```
  здесь мы присваиваем уже существующей переменной `am_isUsed` значение, в зависимости от значения этой же уже существующей переменной `am_isUsed`. Если она равна `false`, ставим `true`, а если она `true`, запускаем функцию `error()`, которая приостанавливает работу STS, формирует красивое сообщение об ошибке, с указанием места срабатывания, стека зон, имени токена и вашим самописным комментарием. 
- `static_modifier_zone` имеет аналогичный механизм, отличается только отсутствием второго токена и изменёным логом ошибки.
Важное замечание, который может возникнуть: что происходит с параметрами зон при заимстовании? Токен `FN` открывает `fun_property`, которая заимствует токены у двух других зон. Токены этих зон опираются на параметры `am_isUsed` и `st_isUsed`. Именно на такие случаи заимстование зон также заимствует параметры.
### Глобальные переменные
На случаи, когда нужно сохранять какие-то глобальные данные, существуют глобальные переменные и флаги – третий вид. К примеру, можно сохранить в переменную токен главной функции, а потом не искать его по всему файлу.
``` java
# объявление глобальной переменной.
$[global_main_fun];
...
FUN {
  ...
  openZone: fun_name;
  # сохраняем текущей токен функции в переменную
  $[fun_token] = $this;
}
FUN_NAME { # токен имени функции
  parentZone: fun_name;
  regex: "/name/";
  ...
  priority: 0;
}
FUN_NAME_MAIN { # токен для имени main, стоящий ваше по приоритету
  parentZone: fun_name;
  regex: "main>/nn/";
  ...
  priority: 1;
  # сохранение fun_token в глобальную переменную
  $global_main_fun = $fun_token
}
```
Здесь я добавил к обычному `FUN_NAME` дополнительный токен `FUN_NAME_MAIN` с приоритетом выше, чем у `FUN_NAME`, за счёт этого, если у функции имя `main`, создаётся токен  `FUN_NAME_MAIN` и переменная `global_main_fun` приравнивается к `fun_token`. В определении `fun_token` встречается упоминание переменной `this` без объявления. Догадливые читатели могли понять, что это ещё одна стандартная переменная, которая ссылается на текущий экземпляр токена. Но в отличии от других стандартных переменных, вроде `text`, её нельзя никак менять.
По итогу при имени `main` переменная `global_main_fun` приравнивается к экземпляру токена функции.
### Типизация
Открытой проблемой становится типизация, которая важна для чёткости и уверености, что код выполнется именно так, как вы хотите.
Тип переменной, это вид данных, которые храняться в переменной. Указывается тип при создании переменной сдедующим образом: `$[имя:тип]`.

| Тип              | Обозначение |
| ---------------- | ----------- |
| Число            | `num`       |
| Строка           | `str`       |
| булевое значение | `bool`      |
| объект           | `obj`       |
| массив           | `arr`       |
Объект – хранит значения, привязанные к текстовым именам.
Массив – хранит значения, привязанные к числовым индексам.
Если не указать тип переменной, то её поведение будет не очень предсказуемым. Тип будет динамически опредяляться и меняться на ходу при потребности.
Чтобы надёжно преобразововать одни типы в другие, используются специальные стандартные функции. Подробнее о функциях будет в разделе «скриптинг».
- `castStrToInt(str)` – преобразует строку из числовых символов в целое число. Кидает ошибку при любых не подходящих символах.
- `castIntToStr(number)` – преобразует целое число в текст. округляет, если число не целое.
С этими знаниями мы вернёмся к sts коду «Анаконды» и внесём некоторые улучшения:
``` java
BODY {
  parentZone: main;
  regex: "\>";
  openZone: body;
  # указываем тип
  $[loops:num] = 1;
}
- body (
  < main;
  < void_zone;
)
BODY_COUNT! {
  parentZone: body;
  regex: "+[0-9]";
  # преобразуем строку в число
  $loops = castStrToInt($text);
}
BODY_CLOSE! {
  parentZone: body;
  regex: ";";
  closeZone;
}
```
Так же нужно разобрать операторы, для работы с разными типами данных.
С числами есть сложение, вычитание, умножение и деление. Работают очень стандартно:
``` java
$[a] = $b + $c;
$a += $b;
$[a] = $b - $c;
$a -= $b;
$[a] = $b * $c;
$a *= $b;
$[a] = $b / $c;
$a /= $b;
```
Со строками действуют три оператора:
- Сложение складывает строки, и приводит к строке другие типы. `$a = "a" + "b"`
- Умножение дублирует строку. `$a *= 2`
- Вычитание позволяет удалить часть символов спереди `$a -= 2`. Если вычесть отричательное число, то будут удалены символы в начале строки.
Для булевых значений есть один отдельный оператор. `!` инвертирует значение. `true` в `false`, а `false` в `true`. Также есть линейная условная конструкция: `condition ? onTrue : onFalse` чтобы встраивать условия прям в вырожения.
Для работы с объектами используется точка. Через неё идёт обращение к полям объекта `object.field`, а далее можно выполнять все стандартные операции. Если в поле лежит число, можно его умножить: `object.number *= 2`.
К элементам массива обращение идёт также через точку. Но кроме этого есть операторы для изменения массива:
- `+` для добавления новых элементов
- `-` для удаления определённого элемента.
- `*` для дублирования элементов
Также к массиву есть ключевой параметр `arr.length` – размер массива.
## setZone
Мы уже давно познакомились с `openZone`, который добавляет новую зону в стек. Это естественный подход при обработке вложенных структур, но иногда нужно изменить контекст парсера, не создавая новых вложеностей; когда синтаксис как-бы последовательный; когда нужно сменить зону после какого-то токена, не делая его родителем. Представлю вам следующий пример:
``` c#
foreach element in list {
  // code...
}
```
Мы имеем ключевое слово `foreach`, после которого нужно написать имя переменной, `in`, имя другой переменной, выполняемый блок кода. `foreach` нужен, чтобы сделать цикл по некоторому массиву. Я бы хотел увидеть примерно такое дерево:
``` yaml
FOREACH:
  IDENTIFIER:
    text: "element"
  IDENTIFIER:
    text: "list"
  BODY: ...
```
В нём нет ничего лишнего. Только два идентификатора и тело цикла. Начнём писать код с очевидного:
``` java
FOREACH {
  parentZone: supports_foreach_zone;
  regex!: "foreach/nn/";
  openZone: foreach_head_zone;
}
- foreach_head_zone (
  < void_zone;
)
```
Но дальше, если мы просто сделаем токены `IDENTIFIER`, `IN`, `FOREACH_BODY_OPEN`, то получиться небезопасный ast, требующий дополнительной синтаксической проверки. А задача sts, предупредить такие ситуации, не дав спарситься неправильному файлу. По этому построим следующий алгоритм действий: 
- читаем токен `IDENTEFIER`
- ожидаем токен `IN`
- читаем токен `IDENTEFIER`
- и наконец ищем `{`
Но как реализовать такой последовательный поиск токенов? Для этого в sts есть команда `setZone`. Она меняет текущую зону, не увеличивая стек. 
``` java
FOREACH_IDENTIFIER_1 {
  parentZone: foreach_head_zone;
  regex: "/name/";
  setZone: foreach_head_zone_1;
}
- foreach_head_zone_1 (<voidZone;)

FOREACH_IN! {
  parentZone: foreach_head_zone_1;
  regex!: "in/nn/";
  setZone: foreach_head_zone_2;
}

- foreach_head_zone_2 (<voidZone;)
FOREACH_IDENTIFIER_2 {
  parentZone: foreach_head_zone_2;
  regex: "/name/";
  setZone: foreach_head_zone_end;
}

- foreach_head_zone_end (<voidZone;)
FOREACH_BODY_OPEN! {
  parentZone: foreach_head_zone_end;
  regex!: "\{";
  setZone: code_zone;
}
```
Здесь токены, начиная с `FOREACH_IDENTIFIER_1` последовательно меняют зону, тем самым реализуя своеобразный конвейер токенов. Самое интересное, что такой подход спокойно поддаётся модифицированию, например созданию ветвлений. Но перед тем как идти дальше, посмотрим, какой ast получился. Для тестов временно добавим код в тело цикла:
``` c#
foreach element in list {
  a a a
}
```
И добавим немного в sts:
``` java
- code_zone (<void_zone;)
A {
  parentZone: code_zone;
  regex!: "a";
}
BODY_CLOSE! {
  parentZone: code_zone;
  regex!: "\}";
}
```
Теперь запустим парсер, после чего получим следующий ast:
``` yaml
FOREACH:
  FOREACH_IDENTIFIER_1:
    text: "element"
  FOREACH_IDENTIFIER_2:
    text: "list"
  A
  A
  A
```
Это не совсем то, что мы хотели получить, потому что код из фигурных скобок находиться в одной куче с двумя идентификаторами. Конечно при пост обработке можно их разделять, но хотелось бы всё таки получить то, что мы хотели получить изначально. Это парсер должен подстраиваться под нас, а не мы под него!
## Комбинирование команд и rename
Для того, чтобы исправить проблему предыдущей темы, нам понадобится научиться комбинировать `openZone`, `setZone` и `closeZone`. Да, вы можете разместить несколько команд в одном токене! Команды обрабатываются в следующем порядке: `setZone > closeZone > openZone`. Из этого вытекают некоторые приятные комбинации:
- `closeZone + openZone` – сначало закрывается текущая зона, а значит список детей токена, открывшего зону, окончательно сформирован. Но после этого выполняется `openZone`, и добавляет в стек новую зону. Смотря на стек зон кажется, что мы просто выполнили `setZone`. Но фокус в том, что последующие токены будут относится к другому токену.
- `setZone + openZone` – ещё одна интересная комбинация. Заменяется текущая зона и сразу открывается новая. Получается, что когда зона закроется, мы будем уже не в той зоне, в которой были до открытия.
Давайте теперь на практике применим оба паттерна.
### Пример 1
Для начала попробуем исправить пример про `foreach`. Вместо `setZone: code_zone;` будем использовать комбинацию `closeZone + openZone`. Если мы просто откроем `code_zone` без `closeZone`, стек будет выглядеть следующим образом:
``` yaml
...
foreach_head_zone_end
code_zone
```
А потом, при встрече `}` нам придётся каким-то образом закрывать сразу две зоны. Можно сделать это костылём:
``` java
CODE_CLOSE! {
  parentZone: code_zone;
  regex!: ">\}";
  closeZone;
}
FOREACH_CLOSE! {
  parentZone: foreach_head_zone_end;
  regex!: "\}";
}
```
Такой костыль использует просмотр вперёд, читая один и тот же текст дважды, чтобы на него сработало два `closeZone`. Но это не чем сложнее будет ваш синтаксис, тем сложнее будет создавать и поддерживать такие костыли, по этому поступим аккуратнее:
``` java
FOREACH_BODY_OPEN {
  parentZone: foreach_head_zone_end;
  regex!: "\{";
  closeZone;
  openZone: code_zone;
}
CODE_CLOSE! {
  parentZone: code_zone;
  regex!: "\}";
}
```
Теперь зона `foreach_head_zone_end` закроется сразу, но токен `FOREACH_BODY_OPEN` откроет `code_zone`, благодаря чему мы всё равно прочитаем код, а когда мы дойдём до закрывающего токена, закроем `code_zone` и вернёмся в доменную зону, не оставив в стеке не следа от `foreach`. Полученный ast:
``` java
FOREACH:
  FOREACH_IDENTIFIER_1:
    text: "element"
  FOREACH_IDENTIFIER_2:
    text: "list"
  FOREACH_BODY_OPEN:
    children:
      A
      A
      A
```
Присмотритесь: это не совсем то дерево, которое мы хотели получить. Проблема в том, что мы хотели совсем другие имена токенов. Почему же я не сделал другие имена токенов? Ну, вероятно где-то в коде уже есть токен `IDENTIFIER` и `BODY`, по этому пришлось делать свои токены, которые вызывают свои команды. Но есть способ это исправить. Параметр `raname` переименовывает токен после создания. Добавим его в наш код, и по итогу получим следующий код:
``` java
FOREACH {
  parentZone: supports_foreach_zone;
  regex!: "foreach/nn/";
  openZone: foreach_head_zone;
}
- foreach_head_zone (<void_zone;)

FOREACH_IDENTIFIER_1 {
  parentZone: foreach_head_zone;
  regex: "/name/";
  setZone: foreach_head_zone_1;
  rename: IDENTIFIER;
}
- foreach_head_zone_1 (<voidZone;)

FOREACH_IN! {
  parentZone: foreach_head_zone_1;
  regex!: "in/nn/";
  setZone: foreach_head_zone_2;
}

- foreach_head_zone_2 (<voidZone;)
FOREACH_IDENTIFIER_2 {
  parentZone: foreach_head_zone_2;
  regex: "/name/";
  setZone: foreach_head_zone_end;
  rename: IDENTIFIER;
}

- foreach_head_zone_end (<voidZone;)
FOREACH_BODY_OPEN {
  parentZone: foreach_head_zone_end;
  regex!: "\{";
  closeZone;
  openZone: code_zone;
  rename: BODY;
}
```
Теперь ast именно такой, какой мы и хотели:
``` java
FOREACH:
  IDENTIFIER:
    text: "element"
  IDENTIFIER:
    text: "list"
  BODY:
    children:
      A
      A
      A
```
### Пример 2
Теперь попробуем скомбинировать `setZone` и `openZone`. А для этого я придумал следующий пример:
``` c#
foreach (element in list) {
  a a a
}
```
Да, здесь я решил добавить обязательные круглые скобки. Также я решил, что теперь хочу в дереве создать отдельный узел `HEAD` для идентификаторов.
Когда они откроются, мы будем читать идентификаторы, но после закрытия мы будем ожидать фигурную скобку. Значит должен сменяться контекст. Но закрывающая круглая скобка не сможет без костылей закрыть и сменить зону. Напомню, что `setZone` обрабатывается ДО `closeZone`. По этому предлагаю выполнять комбинацию команд в `(`. Вот так я переписал код:
``` java
FOREACH {
  parentZone: supports_foreach_zone;
  regex!: "foreach/nn/";
  openZone: foreach_wait_head_zone;
}
- foreach_wait_head_zone (<void_zone;)

FOREACH_HEAD {
  parentZone: foreach_wait_head_zone;
  regex!: "\(";
  # именно тут использую две команды сразу
  setZone: foreach_wait_body_zone;
  openZone: foreach_head_zone_1;
  rename: HEAD;
}
  
FOREACH_IDENTIFIER_1 {
  parentZone: foreach_head_zone_1;
  regex: "/name/";
  setZone: foreach_head_zone_2;
  rename: IDENTIFIER;
}
- foreach_head_zone_in (<voidZone;)

FOREACH_IN! {
  parentZone: foreach_head_zone_in;
  regex!: "in/nn/";
  setZone: foreach_head_zone_2;
}

- foreach_head_zone_2 (<voidZone;)
FOREACH_IDENTIFIER_2 {
  parentZone: foreach_head_zone_2;
  regex: "/name/";
  setZone: foreach_head_zone_end;
  rename: IDENTIFIER;
}

- foreach_head_zone_end (<voidZone;)
FOREACH_HEAD_END! {
  parentZone: foreach_head_zone_end;
  regex!: "\)";
  closeZone;
}

- foreach_wait_body_zone (<voidZone;)
FOREACH_BODY {
  parentZone: foreach_wait_body_zone;
  regex!: "\{";
  openZone: code_zone;
  remame: BODY;
}
```
## Паттерны
В предыдущей теме я один раз использовал слово «паттерн». Люди используют множество разных паттернов в sts, и я думаю, что с развитием языка их будет становиться всё больше.
Паттерн – это не какой-то особый элемент языка, это просто подход созданный пользователями на основе базовых функций. И в этой главе я расскажу о части известных мне паттернов.
### Конвейер
При реализации `foreach` мы использовали множество `setZone`. Это называется конвейером. Подход с `setZone` хорош, когда синтаксис может иметь ветвления. Например можно было бы добавить конструкцию `from ... to ...`, которую можно было бы использовать на ровне с `in`. Также `setZone` может обеспечить перечисления, заранее неизвестного размера. Но если мы заранее точно знаем, последовательность каких зон мы ожидаем, то можно использовать более аккуратный способ, основанный на многократном `openZone`.
``` c#
foreach element in list {
  a a a
}
```
Всё тот-же код. Но реализую я его по другому.
``` java
FOREACH {
  parentZone: supports_foreach_zone;
  regex!: "foreach/nn/";
  openZone:
    foreach_head_zone_1,
    foreach_head_zone_in,
    foreach_head_zone_2,
    foreach_wait_body_zone;
}
~ void_zone (
  > foreach_head_zone_1;
  > foreach_head_zone_in;
  > foreach_head_zone_2;
  > foreach_wait_body_zone;
)

FOREACH_IDENTIFIER_1 {
  parentZone: foreach_head_zone_1;
  regex: "/name/";
  closeZone;
  rename: IDENIFIER;
}
FOREACH_IN! {
  parentZone: foreach_head_zone_in;
  regex!: "in/nn/";
  closeZone;
}
FOREACH_IDENTIFIER_2 {
  parentZone: foreach_head_zone_2;
  regex: "/name/";
  closeZone;
  rename: IDENIFIER;
}
FOREACH_BODY {
  parentZone: foreach_wait_body_zone;
  regex!: "\{";
  closeZone;
  openZone: code_zone;
  rename: BODY;
}
```
Да, такой пример намного читабельней, но что здесь вообще происходит!?
Давайте по порядку:
- `openZone` – данный параметр можно нагрузить списком элементов также, как и `parentZone`. Работает это так: зоны открываются по очереди, после того как предыдущая закрывается. Все токены из всех зон последовательно объединяются в `children`. (Если одна зона закрылась, но её последний токен использовал `closeZone + openZone:abc`, то следующая зона откроется только после того, как закроется подзона `abc` того токена.) Дальше мы делаем токены в каждой зоне, вызывающие `closeZone`, за счёт чего после каждого токена начинается новая зона. Так вся логика смены зон находится в одном месте, а не размазана по объявлениям токенов.
- `~ void_zone` – оператор `~` нужен, чтобы модифицировать уже существующие токены и зоны. Про модифицирование будет следующая глава. В крации, я просто добавляю в `void_zone` обратные заимствования, благодаря чему, нам не приходиться делать множество объявлений зон, и не приходиться менять исходное объявление `void_zone`.
Немного подумав, я пришёл к выводу, что данный подход спокойно можно совместить с `setZone`, и этому нет никаких помех. Вы можете переключаться между скобками `foreach (...) {...}` через многократный `openZone`, а в рамках отдельных элементов реализововать через череду `setZone`.

## namespaces
Чтобы ваш sts файл не выглядел как свалка, существуют способы группировки объявлений в логические секторы – неймспейсы. Для объявления используется знак `%`. Неймспейсы можно вкладывать в друг друга. В рамках одного неймспейса можно обращаться к соседним именам без каких либо проблем. А для обращения к элементу в другом namespace используется точка. Пример:
``` java
% main (
  % code (
    - code_zone ();
    CODE_CLOSE {
      # локальное обращение
      parentZone: code_zone;
      regex: "\}";
      clo‌seZone;
    }
  )
  CODE {
    parentZone: code_open_zone;
    regex: "\{";
    # относительное обращение
    openZone: code.code_zone;
  }
)
% foreach {
  ...
  FOREACH_BODY_OPEN {
    parentZone: foreach_wait_body_zone;
    regex: "\{";
    # глобальное обращение
    openZone: main.code.code_zone;
  }
}
```
так же можно создать безымянный неймспейс просто для группировки без усложнения пути до элемента 
``` java

  parentZone: aaa;
  regex: "a";
}

  )
```

## Наследование
Нередки ситуации, когда нам требуется объявить много похожих токенов с небольшими отличиями, но с большим количеством полей. Представим следующую ситуацию:
``` java
ACCESS_MOD_PUBLIC {
  parentZone: access_modifiers;
  regex: "public/nn/";
  rename: ACCESS_MOD;
  $[type] = "public";
  $access_mod_is_used = 
    $access_mod_is_used
    ? error("access modifier already used")
    : true;
}
```
вы успешно написали 10 строк кода, ради модификатора `public`. Осталось повторить это ещё 2 раза, чтобы добавить `protected`, `private`, при этом в каждом сделав всего 2 изменения.
Умный в гору не пойдёт, умный гору обойдёт. Воспользуемся наследованием, которое копирует все параметры из другого объявления:
``` java
ACCESS_MOD_PROTECTED <- ACCESS_MOD_PUBLIC {
  regex: "protected/nn/";
  $[type] = "protected";
}
ACCESS_MOD_PRIVATE <- ACCESS_MOD_PUBLIC {
  regex: "private/nn/";
  $[type] = "private";
}
```
Когда мы скопировали параметры, можно часть из них переопределить, или добавить новые. Ещё пример:
``` java
% func {
  FUNC {
    parentZone: supports_func_zone
    regex: "function/nn/";
    openZone: func_head, func_body_wait;
  }
  ...
}
% class {
  CLASS {
    parentZone: supports_class_zone;
    regex: "class/vvoid/\{";
    openZone: class_zone;
    $[currentClass] = $this;
    $[methods:arr];
  }
  METHOD <- func.FUNC {
    regex: "method/nn/";
    $[parent] = $currentClass;
    $parent.methods += $this;
  }
}
```
